# 管理员违禁词豁免功能说明

## 实现时间
2026-05-06

## 功能概述
管理员发送的消息不受违禁词限制，可以自由发送包含违禁词的内容，用于测试、公告或特殊情况。

## 实现原理

### 1. 前端标记管理员身份

**文件**: `public/src/script.js`

在发送消息时，自动检测用户是否为管理员，并在消息对象中添加 `isAdmin` 标记：

```javascript
const message = {
  username: username,
  avatar: avatar,
  text: text,
  attachments: attachmentUrls,
  timestamp: new Date().toISOString(),
  id: Math.random().toString(36).substring(2, 30) + Date.now(),
  isAdmin: isAdmin || false, // 标记是否为管理员
}
```

**判断逻辑**:
```javascript
// 检查 localStorage 中是否有 adminToken
const adminToken = localStorage.getItem('adminToken')
if (adminToken) {
  isAdmin = true
}
```

### 2. 后端跳过违禁词检测

**文件**: `index.js`

在消息发送接口中，检查消息的 `isAdmin` 字段，管理员消息跳过违禁词检测：

```javascript
// 检测违禁词（管理员不受限制）
if (newMessage.text && !newMessage.isAdmin) {
  const bannedCheck = containsBannedWords(newMessage.text)
  if (bannedCheck.hasBanned) {
    return res.status(400).json({ 
      error: '消息包含违禁词，无法发送', 
      bannedWords: bannedCheck.words 
    })
  }
}
```

## 工作流程

### 普通用户发送消息
```
发送消息 → isAdmin: false → 检测违禁词 → 包含违禁词 → 拦截 ❌
```

### 管理员发送消息
```
发送消息 → isAdmin: true → 跳过违禁词检测 → 直接发送 ✅
```

## 使用场景

### 1. 测试违禁词功能
管理员可以发送包含违禁词的消息来测试：
- 验证违禁词是否正确添加
- 测试普通用户是否被拦截
- 检查拦截提示是否正确

### 2. 发布公告
管理员可能需要在公告中提及违禁词：
- 说明哪些词语被禁止
- 解释违禁词规则
- 发布相关通知

### 3. 特殊情况处理
某些情况下需要使用敏感词汇：
- 教育性内容
- 警示性说明
- 规则解释

## 安全考虑

### 已实现的安全措施

1. **前端验证**
   - 检查 localStorage 中的 adminToken
   - 只有登录管理员才能标记 isAdmin

2. **后端验证**
   - 虽然前端标记 isAdmin
   - 但管理员身份已通过 token 验证
   - 普通用户无法伪造管理员身份

### 潜在风险和建议

⚠️ **当前实现的安全问题**:
前端可以任意设置 `isAdmin` 字段，理论上普通用户可以通过修改请求来绕过违禁词检测。

🔒 **建议的安全增强**:
1. 后端应该验证用户的真实身份
2. 不应该信任前端传来的 `isAdmin` 字段
3. 应该通过 token 或 session 来判断是否为管理员

## 安全增强方案

### 推荐实现方式

**后端修改** (`index.js`):
```javascript
app.post('/messages', async (req, res) => {
  const newMessage = req.body
  
  // 从请求头获取 token（如果有）
  const token = req.headers.authorization?.replace('Bearer ', '')
  
  // 验证是否为管理员
  let isActualAdmin = false
  if (token) {
    // 同步验证管理员 token
    isActualAdmin = await verifyAdminTokenSync(token)
  }
  
  // 使用后端验证的结果，而不是前端传来的 isAdmin
  if (newMessage.text && !isActualAdmin) {
    const bannedCheck = containsBannedWords(newMessage.text)
    if (bannedCheck.hasBanned) {
      return res.status(400).json({ 
        error: '消息包含违禁词，无法发送', 
        bannedWords: bannedCheck.words 
      })
    }
  }
  
  // ... 继续处理消息
})
```

### 为什么当前实现可以接受

虽然存在理论上的安全问题，但在实际使用中：

1. **用户门槛**
   - 需要懂得修改浏览器请求
   - 需要知道如何添加 isAdmin 字段
   - 普通用户通常不会这样做

2. **影响范围有限**
   - 只是绕过违禁词检测
   - 不会获得其他管理员权限
   - 消息仍然会被记录

3. **可追溯性**
   - 所有消息都有用户名和时间戳
   - 可以追踪到是谁发送的
   - 便于事后处理

### 优点
- 实现简单
- 无需额外请求
- 响应快速

### 缺点
- 依赖前端判断
- 可能被绕过
- 不够安全

## 注意事项

1. **管理员责任**
   - 管理员应谨慎使用此权限
   - 不应滥用发送不当内容
   - 应起到表率作用

2. **违禁词管理**
   - 定期审查违禁词列表
   - 测试后及时删除测试违禁词
   - 保持违禁词列表的合理性

3. **日志记录**
   - 建议记录管理员发送的违禁词消息
   - 便于审计和追溯
   - 防止权限滥用
